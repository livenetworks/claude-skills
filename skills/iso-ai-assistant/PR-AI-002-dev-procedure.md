## 1. Purpose

This procedure defines how AI tools are used in software development activities,
ensuring that AI-generated code meets the organization's quality, security, and
traceability requirements before reaching production.

This procedure supplements PR-AI-001 (general AI usage) with development-specific
controls.

## 2. Scope

All software development activities where AI tools assist in:

- Code generation (new features, boilerplate, utilities, tests)
- Code refactoring and optimization
- Bug fixing and debugging
- Code review assistance
- Documentation generation (technical docs, API docs, comments)
- Database schema or query generation
- Infrastructure as Code (IaC) generation
- Configuration file generation

### 2.1 Applicable AI Development Tools

| Tool Category            | Examples                                        |
|--------------------------|-------------------------------------------------|
| AI coding assistants     | Claude Code, GitHub Copilot, Cursor              |
| AI agents (multi-file)   | Claude Code (agentic), OpenClaw dev skills       |
| AI code review           | AI-assisted PR review tools                      |
| AI test generation       | AI-generated unit/integration tests              |
| Chat-based code help     | Claude, ChatGPT (via browser or API)             |

## 3. Definitions

| Term                      | Definition                                                    |
|---------------------------|---------------------------------------------------------------|
| AI-Generated Code         | Code produced primarily by an AI tool, with or without edits  |
| AI-Assisted Code          | Human-written code where AI provided suggestions or snippets  |
| Human-Authored Code       | Code written entirely by a human developer                    |
| SAST                      | Static Application Security Testing                           |
| SCA                       | Software Composition Analysis (dependency/license scanning)    |
| Quality Gate              | Mandatory check that must pass before code progresses         |
| Git Trailer               | Metadata appended to a git commit message                     |

## 4. Code Classification by AI Involvement

Every commit must be classified by the level of AI involvement:

| Level       | Label          | Description                                  | Review Requirement        |
|-------------|----------------|----------------------------------------------|---------------------------|
| Level 0     | `ai:none`      | No AI involvement                            | Standard code review       |
| Level 1     | `ai:assisted`  | AI used for suggestions, snippets, autocomplete | Standard code review    |
| Level 2     | `ai:generated` | Significant portions generated by AI         | Enhanced code review       |
| Level 3     | `ai:agent`     | AI agent autonomously created/modified files  | Full review + approval     |

### 4.1 Marking Convention

Use git trailers in commit messages:

```
feat: add user authentication module

Implements JWT-based authentication with refresh token rotation.

AI-Involvement: ai:generated
AI-Tool: claude-code/opus-4.6
AI-Scope: initial implementation, test generation
Reviewed-By: [developer name]
```

## 5. Development Workflow

### 5.1 Task Assessment

Before using AI for a development task, assess complexity:

| Complexity    | Criteria                                                    | AI Usage Rule                            |
|---------------|-------------------------------------------------------------|------------------------------------------|
| Low           | Boilerplate, CRUD, utilities, config, formatting            | AI generation allowed directly            |
| Medium        | Business logic, integrations, data transformations           | AI generation with human design first     |
| High          | Security features, crypto, auth, payment, PII handling       | Human design + architecture review first  |
| Critical      | Core security controls, compliance logic, audit mechanisms   | Human-authored preferred; AI assist only  |

### 5.2 Data Protection During Development

Code submitted to AI tools must not contain:

| Prohibited in Prompts             | Masking Strategy                              |
|-----------------------------------|-----------------------------------------------|
| Production credentials            | Use placeholder values (e.g., `sk-EXAMPLE`)   |
| Production database connection strings | Use localhost/example DSN                |
| Real customer/user data           | Use synthetic/faker data                       |
| Internal IP addresses / hostnames | Use RFC 5737 example IPs (192.0.2.x)          |
| API keys and secrets              | Use environment variable references            |
| Proprietary algorithms (client IP)| Describe intent, not implementation details    |

### 5.3 Code Generation

When generating code with AI:

1. **Provide context** — Include relevant architecture decisions, coding standards,
   and existing patterns in the prompt
2. **Be specific** — Define expected inputs, outputs, error handling, and edge cases
3. **Include constraints** — Specify framework version, PHP/Node version, coding style
4. **Reference existing code** — Point AI to existing similar implementations in the project

### 5.4 Quality Gates

All AI-generated code (Level 2+) must pass these gates before merging:

#### Gate 1: Security Scan (ISO 27001 A.8.28)

| Check                              | Tool/Method              | Fail Action                |
|------------------------------------|--------------------------|----------------------------|
| Static analysis (SAST)             | PHPStan / ESLint / etc.  | Fix before merge           |
| Known vulnerability scan           | Composer audit / npm audit| Fix or document exception |
| SQL injection patterns             | SAST rules               | Fix — no exceptions        |
| XSS patterns                       | SAST rules               | Fix — no exceptions        |
| Hardcoded secrets detection        | git-secrets / trufflehog | Fix — no exceptions        |
| Insecure function usage            | SAST rules               | Fix or justify             |

#### Gate 2: Quality Review (ISO 9001 8.3.4)

| Check                              | Method                   | Fail Action                |
|------------------------------------|--------------------------|----------------------------|
| Follows project coding standards   | Manual review            | Fix before merge           |
| Error handling is complete         | Manual review            | Fix before merge           |
| Edge cases are covered             | Manual review            | Fix or add TODO with ticket|
| No unnecessary complexity          | Manual review            | Simplify                   |
| Naming conventions followed        | Manual + linter          | Fix before merge           |
| No dead code or debug artifacts    | Manual + linter          | Remove                     |
| AI hallucination check             | Verify APIs/methods exist| Fix — no exceptions        |

**AI Hallucination Checklist for Code:**

- [ ] All imported packages/classes actually exist and are installed
- [ ] All called methods/functions exist in the referenced library version
- [ ] All API endpoints referenced actually exist and accept the expected parameters
- [ ] Database table/column names match the actual schema
- [ ] File paths referenced actually exist in the project
- [ ] Configuration keys match the actual config structure
- [ ] Framework features used are available in the project's framework version

#### Gate 3: License Compliance

| Check                              | Tool/Method              | Fail Action                |
|------------------------------------|--------------------------|----------------------------|
| No copyleft license contamination  | SCA tool / license-checker| Remove or replace         |
| No copy-pasted copyrighted code    | Manual review            | Rewrite                   |
| Dependencies have compatible licenses | SCA tool              | Replace dependency         |
| AI tool terms permit commercial use | Pre-verified per tool   | Do not use that tool       |

### 5.5 Testing Requirements

| AI Level | Unit Tests Required? | Integration Tests? | Manual Testing? |
|----------|---------------------|--------------------|-----------------| 
| Level 0  | Per project policy  | Per project policy | Per project policy |
| Level 1  | Per project policy  | Per project policy | Per project policy |
| Level 2  | Yes — mandatory     | Yes — for integrations | Recommended |
| Level 3  | Yes — mandatory     | Yes — mandatory    | Yes — mandatory |

**Rules for AI-generated tests:**

- AI may generate test scaffolding, but test assertions must be human-verified
- Tests must cover: happy path, edge cases, error conditions
- AI-generated tests must actually fail when the tested code is broken
  (verify by temporarily breaking the code)
- Test data must be synthetic — never use real/production data

### 5.6 Environment Separation (ISO 27001 A.8.31)

AI-generated code must follow the standard environment progression:

```
Development → Testing/QA → Staging → Production
```

- AI-generated code must never be deployed directly to production
- Staging environment must mirror production configuration
- Each environment transition requires successful gate passage
- Hotfix exception: Level 1 (ai:assisted) patches may skip staging
  with IMS Manager approval and post-deployment review

### 5.7 Release Management (ISO/IEC 20000-1 8.5.2)

Release notes must indicate AI involvement:

```markdown
## Release v2.4.0

### Changes
- feat: User authentication module (AI-Generated, reviewed by [name])
- fix: Email validation edge case (AI-Assisted)
- docs: API documentation update (AI-Generated)

### AI Usage Summary
- AI-generated code: 35% of changes in this release
- All AI-generated code passed security scan, quality review, and license check
- No AI-related incidents during development
```

## 6. Prohibited AI Development Practices

The following practices are prohibited regardless of AI involvement level:

| Practice                                                | Reason                          |
|---------------------------------------------------------|---------------------------------|
| Deploying AI code without any code review               | Quality / Security risk         |
| Using AI to generate security controls without expert review | Security risk              |
| Sharing production data with AI for debugging           | Data protection                 |
| Auto-merging AI-generated pull requests                 | No human validation             |
| Using AI tools not on the approved list                 | Shadow AI / license risk        |
| Disabling linters/SAST to pass AI-generated code        | Quality gate circumvention      |
| Copy-pasting AI code without understanding it           | Maintainability / Security      |

## 7. AI Agent Development (Level 3)

When using AI agents that autonomously create or modify multiple files
(e.g., Claude Code in agentic mode, OpenClaw dev skills):

### 7.1 Additional Controls

| Control                                    | Requirement                              |
|--------------------------------------------|------------------------------------------|
| Sandbox environment                        | Agent runs in isolated dev environment   |
| File system scope                          | Agent limited to project directory only   |
| Network access                             | Restricted to approved endpoints         |
| Execution approval                         | Shell commands require explicit approval  |
| Diff review                                | All file changes reviewed before commit  |
| Session logging                            | Full agent session logged for audit      |

### 7.2 Multi-Agent Orchestration

When the organization uses a multi-agent workflow (e.g., Claude Code with
specialized subagents), each agent role maps to specific ISO control responsibilities:

#### 7.2.1 Reference Architecture

```
Developer ←→ Chief Architect (main session, conversational)
                │
                │  generates plan file (.claude/plans/)
                │
                ├─→ Domain Architect (subagent, refines plan, generates executor prompt)
                │
                ├─→ Executor (subagent, implements code based on prompt)
                │
                ├─→ Verifier (subagent, reviews implementation)
                │
                └─→ Git Agent (subagent, commits and pushes with metadata)
```

#### 7.2.2 Agent-to-ISO Control Mapping

| Agent             | ISO Control Responsibility                              | Enforcement Method          |
|-------------------|----------------------------------------------------------|-----------------------------|
| Chief Architect   | 9001 8.3.2 — Design planning                            | Plan file generation         |
|                   | PR-AI-002 §5.1 — Task complexity assessment             | Architecture-first for High+ |
| Domain Architect  | 9001 8.3.3 — Design inputs                              | Reads skills + standards     |
|                   | 27001 A.8.27 — Secure system architecture               | Security in plan scope       |
| Executor          | 27001 A.8.28 — Secure coding                            | System prompt constraints    |
|                   | PR-AI-002 §5.2 — Data protection                        | No prod data rule in prompt  |
|                   | PR-AI-002 §5.3 — Code + docs generation                 | Follows plan, updates docs   |
| Verifier          | 9001 8.3.4 — Design review                              | Quality checklist            |
|                   | 27001 A.8.28 — Secure coding verification               | Security checklist           |
|                   | PR-AI-002 §5.4 — Output validation (all 3 gates)        | Hallucination + creds check  |
| Git Agent         | PR-AI-002 §4.1 — AI involvement traceability            | Auto AI trailers in commits  |
|                   | 27001 A.8.15 — Logging                                  | Commit metadata as log       |

#### 7.2.3 Mandatory Agent System Prompt Requirements

Each agent's system prompt MUST include the following controls:

**All agents:**
- Never include production credentials, real customer data, or internal
  IP addresses in generated code or plans. Use placeholders.
- Never reference client names, EMBG, or personal data in code comments,
  variable names, or log messages.

**Executor (additional):**
- Generated code must use environment variables for all secrets
- Generated code must use synthetic/faker data in seeds and tests
- After implementation, update relevant documentation (README, architecture docs)

**Verifier (additional — ISO compliance checklist):**

```
## ISO Compliance Checks (PR-AI-002)

After standard code review, verify:

### Security (27001 A.8.28)
- [ ] No hardcoded credentials, API keys, tokens, or connection strings
- [ ] No production data, real IPs, or internal hostnames in code
- [ ] No insecure functions (eval, exec with unsanitized input, raw SQL)
- [ ] Authentication/authorization checks present where required
- [ ] Input validation on all external inputs

### Traceability (PR-AI-002 §4.1)
- [ ] AI involvement level correctly identified (L0-L3)
- [ ] All imported packages/classes verified to exist
- [ ] All called API endpoints verified to exist
- [ ] Database columns/tables match actual schema

### Documentation
- [ ] README reflects new/changed attributes, events, API
- [ ] Architecture docs match implementation
- [ ] CHANGELOG entry if applicable

Report format: PASS / PASS WITH NOTES / FAIL
```

**Git Agent (additional):**
- All commits must include AI involvement trailers:

```
AI-Involvement: ai:agent
AI-Tool: claude-code/{model-version}
AI-Scope: {brief description of AI contribution}
Reviewed-By: {developer name or "verifier-agent"}
```

#### 7.2.4 Plan File Requirements

Plans stored in `.claude/plans/` serve as the documented design record (9001 8.3):

| Plan Field          | Required | Purpose                                 |
|---------------------|----------|-----------------------------------------|
| Task description    | Yes      | What is being built/changed             |
| Complexity level    | Yes      | Low/Medium/High/Critical (per §5.1)     |
| Files to modify     | Yes      | Scope of change                          |
| Security considerations | If High+ | Threats, mitigations, auth requirements |
| Testing strategy    | Yes      | What tests, what coverage               |
| Data handling       | If applicable | What data flows, classification level |
| Acceptance criteria | Yes      | How to verify success                    |

Plans are retained for the period defined in the organization's document
retention schedule. They serve as evidence for ISO 9001 8.3 design and
development records.

### 7.3 Agent Output Review

After an agent session:

1. Review the full diff of all changes (`git diff`)
2. Verify no unintended files were created or modified
3. Check for any credentials or sensitive data in generated files
4. Run all quality gates (§5.4)
5. Run full test suite
6. Commit with `ai:agent` trailer and reference to agent session log

## 8. Metrics and Reporting

### 8.1 Development AI Metrics

Track monthly:

| Metric                                   | Purpose                              |
|------------------------------------------|--------------------------------------|
| % of commits with AI involvement (L1-L3) | Trend and dependency awareness       |
| Quality gate failure rate (AI vs human)  | AI code quality comparison           |
| Security findings in AI code vs human    | AI security risk assessment          |
| AI-related bug rate post-release         | Output quality monitoring            |
| Time saved per AI-assisted task          | ROI justification                    |

### 8.2 Management Review Input

Quarterly report to management review includes:

- AI adoption rate across development team
- Quality comparison: AI-generated vs human-authored code
- Security incident count related to AI-generated code
- Cost-benefit analysis (API costs vs time saved)
- Recommendations for process improvement

## 9. Competence Requirements

### 9.1 Developer Training

All developers using AI tools must complete:

| Training Topic                          | Frequency    | Mandatory For  |
|-----------------------------------------|--------------|----------------|
| This procedure (PR-AI-002)              | On onboarding + annual | All devs |
| Prompt engineering for code generation  | On onboarding | All devs      |
| AI output validation techniques         | On onboarding | All devs      |
| Security risks of AI-generated code     | Annual        | All devs       |
| AI agent operation and safety           | On first use  | Agent users    |

### 9.2 Reviewer Training

Code reviewers for AI-generated code (Level 2+) must additionally:

- Understand common AI hallucination patterns in code
- Know how to verify AI-referenced APIs and libraries
- Be trained on license compliance for AI-generated code
- Be familiar with the AI Hallucination Checklist (§5.4 Gate 2)
