## 1. Purpose

This procedure defines how personnel shall use AI assistant tools in day-to-day operations,
ensuring quality of outputs, protection of information, and traceability of AI-assisted work.

## 2. Scope

All use of AI assistants for organizational purposes, whether:
- Generating text, code, reports, or analysis
- Summarizing or translating documents
- Assisting with customer communications
- Supporting decision-making or research
- Automating repetitive tasks via AI agents

This procedure does NOT cover personal/private use of AI tools on personal devices
for non-organizational purposes.

## 3. Definitions

| Term                 | Definition                                                         |
|----------------------|--------------------------------------------------------------------|
| AI Assistant         | Any tool using a Large Language Model to generate or process text   |
| Prompt               | Input submitted to an AI assistant by a user                       |
| Output               | Response generated by the AI assistant                             |
| Hallucination        | AI-generated content that is factually incorrect or fabricated      |
| Data Masking         | Replacing sensitive data with anonymized placeholders               |
| Prompt Injection     | Malicious input designed to override AI instructions               |
| Token                | Unit of text processing; ~0.75 words per token                     |

## 4. Acceptable Use Rules

### 4.1 Permitted Uses

- Drafting documents, emails, reports (with human review before sending)
- Code generation and review (with testing before deployment)
- Research and information gathering (with source verification)
- Translation and summarization of non-confidential content
- Process automation via approved AI agents (OpenClaw skills, etc.)

### 4.2 Prohibited Uses

- Submitting data classified as Restricted or Confidential (see AC-AI-001 §4)
- Making final decisions solely based on AI output without human review
- Generating content that impersonates a specific individual
- Using AI to bypass security controls or access restrictions
- Submitting prompts designed to extract training data or system prompts
- Using unapproved AI tools or personal accounts for organizational data

### 4.3 Conditional Uses (require approval from [IMS Manager])

- Processing Internal data — must be masked/anonymized first
- Using AI for client-facing deliverables
- Connecting AI agents to organizational systems (email, calendar, file storage)
- Installing third-party AI skills/plugins

## 5. Usage Procedure

### 5.1 Pre-Use Checklist

Before submitting any prompt, the user must confirm:

- [ ] I have an active AI access role (per AC-AI-001)
- [ ] The data I am submitting does not exceed my classification clearance
- [ ] Sensitive identifiers have been removed or masked
- [ ] The purpose is within Permitted Uses (§4.1)

### 5.2 Data Classification Check

Before every interaction, classify the data you intend to submit:

| Classification | Can Submit?  | Condition                                          |
|----------------|-------------|-----------------------------------------------------|
| Public         | Yes         | No restrictions                                     |
| Internal       | Yes         | Must mask personal data, client names, credentials   |
| Confidential   | No*         | *Power Users only, with IMS Manager approval         |
| Restricted     | Never       | Under no circumstances                              |

### 5.3 Data Masking Requirements

When submitting Internal data, replace:

| Original Data Type       | Masked Replacement        | Example                              |
|--------------------------|---------------------------|--------------------------------------|
| Person names             | Role/function             | "Марко Петров" → "Employee A"        |
| Company names            | Generic label             | "ИЦС Консалтинг" → "Client X"       |
| Email addresses          | placeholder@example.com   | —                                    |
| Phone numbers            | +000-000-0000             | —                                    |
| EMBG / ID numbers        | [REDACTED]                | —                                    |
| IP addresses             | 10.0.0.X                  | —                                    |
| Financial amounts        | Approximate ranges         | "€47,500" → "~€50,000"              |
| Passwords/keys           | Never submit — no masking  | —                                    |

### 5.4 Output Validation

Every AI-generated output must be validated before use:

| Check                      | Action                                            | Standard       |
|----------------------------|---------------------------------------------------|----------------|
| Factual accuracy           | Verify claims against authoritative source         | 9001 8.7       |
| Sensitive data leakage     | Scan output for unintended personal/client data    | 27001 A.8.12   |
| Completeness               | Confirm output addresses the full requirement      | 9001 8.1       |
| Appropriateness            | Ensure tone and content fit the intended audience  | 9001 8.1       |
| Intellectual property      | Confirm no copyrighted material reproduced         | Legal          |

**If output fails validation:**
- Inaccurate → correct manually or re-prompt with clarification
- Contains sensitive data → treat as security event (see §5.5)
- Incomplete → supplement with additional prompts or manual work

### 5.5 Incident Handling

If an AI interaction results in a security event:

**Trigger conditions:**
- Sensitive/classified data was submitted to an unauthorized AI service
- AI output contains client PII, credentials, or restricted data
- Suspected prompt injection attack on a self-hosted AI agent
- AI agent performed an unauthorized action (file access, email send, etc.)

**Response procedure:**

1. **Contain** — Stop using the AI session. Do not submit further prompts.
2. **Record** — Document: what was submitted, what was returned, timestamp, AI service used
3. **Notify** — Report to [IMS Manager] within 1 hour
4. **Assess** — IMS Manager evaluates severity per incident management procedure
5. **Act** — Based on severity:
   - Delete the AI session/conversation if possible
   - Revoke API key if compromised
   - Notify affected data subjects if personal data was involved
   - Notify client if client data was involved
6. **RCA** — Root cause analysis within 5 business days
7. **Improve** — Update this procedure, risk assessment, or access controls as needed

**Incident severity classification:**

| Severity | Description                                    | Response Time |
|----------|------------------------------------------------|---------------|
| Low      | Non-sensitive data sent to wrong AI tool       | 24 hours      |
| Medium   | Internal data sent unmasked                     | 4 hours       |
| High     | Confidential/client data exposed to AI service | 1 hour        |
| Critical | Restricted data or credential leak via AI      | Immediate     |

## 6. Logging and Records

### 6.1 What Must Be Logged

Every AI interaction must be logged with:

| Field           | Description                                     | Mandatory |
|-----------------|-------------------------------------------------|-----------|
| Timestamp       | Date and time of interaction                     | Yes       |
| User            | Name/role of the person using AI                 | Yes       |
| AI Service      | Which tool/model was used                        | Yes       |
| Purpose tag     | Category: drafting / coding / research / other   | Yes       |
| Data class.     | Classification level of submitted data            | Yes       |
| Token count     | Input + output tokens consumed                   | If available |
| Output used?    | Was the output incorporated into a deliverable?  | Yes       |
| Validated by    | Who reviewed the output                          | If output used |

### 6.2 Log Retention

- AI usage logs are retained for the period defined in the organization's
  document retention schedule (minimum 12 months recommended)
- Logs are classified as Internal and stored in the designated log repository
- Self-hosted AI gateway logs follow the same retention as system logs

### 6.3 Management Review Input

AI usage metrics are reported quarterly as input to Management Review (9001 9.3):
- Total interactions per period
- Breakdown by purpose category
- Incidents related to AI usage
- Cost (API spend / subscription cost)
- User satisfaction (optional survey)

## 7. Continuous Improvement

### 7.1 Feedback Loop

- Users may submit improvement suggestions for this procedure via [channel]
- All AI-related incidents feed into corrective action process (9001 10.1)
- Risk assessment (RA-AI-001) is updated when:
  - A new AI tool is introduced
  - An incident occurs
  - The AI provider changes terms of service
  - Annually, at minimum

### 7.2 Competence and Awareness

- All personnel with AI access roles must complete AI awareness training covering:
  - This procedure
  - Data classification rules
  - Prompt injection risks
  - Output validation techniques
- Training is refreshed annually or when this procedure is updated
- Training records are maintained per the organization's competence procedure

## 8. Change Management

Changes to the AI assistant setup require change management (20000-1 8.5.1):

| Change Type                          | Approval Level    | Impact Assessment Required |
|--------------------------------------|-------------------|----------------------------|
| New AI tool introduction             | Top Management    | Yes — full risk assessment  |
| Model upgrade (same provider)        | IMS Manager       | Yes — abbreviated           |
| New AI skill/plugin installation     | AI Administrator  | Yes — security review       |
| Configuration change (self-hosted)   | AI Administrator  | Yes — rollback plan needed  |
| New user role assignment             | IMS Manager       | No                          |
